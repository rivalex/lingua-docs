# Двунаправленная синхронизация

Lingua может импортировать переводы из локальных файлов в базу данных и экспортировать их обратно — давая вам лучшее из обоих миров: **runtime на основе БД** и **версионный контроль через файлы**.

## Два направления

```
lang/en/*.php        ──┐
lang/en.json           │  lingua:sync-to-database ──►  language_lines (DB)
lang/it/*.php          │
lang/it.json           │
lang/vendor/…          │  ◄── lingua:sync-to-local
                      ─┘
```

## Синхронизация в базу данных

Импортирует все файлы переводов из `lang/` (и его поддиректорий) в таблицу `language_lines`.

```bash
php artisan lingua:sync-to-database
```

### Что импортируется

- `lang/{locale}/*.php` — стандартные PHP-файлы переводов
- `lang/{locale}.json` — JSON-файлы переводов
- `lang/vendor/{package}/{locale}/*.php` — переводы пакетов

### Поведение upsert

Lingua использует `updateOrCreate` с сопоставлением по `group` + `key`. Это означает:
- **Новые ключи** вставляются
- **Существующие ключи** получают слияние JSON `text` — значения локалей, отредактированные в интерфейсе, **сохраняются**
- **Определение типа** запускается по значению для определения `text` / `html` / `markdown`

### Автоопределение типа

| Правило | Назначаемый тип |
|---|---|
| Строка содержит HTML-теги (`<…>`) | `html` |
| Строка парсится как Markdown (заголовки, списки и т.д.) | `markdown` |
| Ни то, ни другое | `text` |

::: tip
Определение типа консервативно — `html` или `markdown` назначается только когда содержимое явно соответствует. Простые строки всегда получают тип `text`. Вы можете изменить тип вручную через модальное окно редактирования.
:::

### Через фасад

```php
use Rivalex\Lingua\Facades\Lingua;

Lingua::syncToDatabase();
```

::: warning
Фасад вызывает `Translation::syncToDatabase()` внутренне, что является статическим вызовом. В Artisan-командах и Livewire-компонентах вместо этого используется `app(Translation::class)->syncToDatabase()`, чтобы Mockery мог перехватить его в тестах.
:::

## Синхронизация в локальные файлы

Экспортирует все переводы из `language_lines` обратно в PHP и JSON файлы в `lang/`.

```bash
php artisan lingua:sync-to-local
```

### Что экспортируется

- Все не-vendor переводы → `lang/{locale}/{group}.php`
- Ключи JSON-группы (`single`) → `lang/{locale}.json`
- Переводы пакетов → `lang/vendor/{vendor}/{locale}/{group}.php`

### Сценарии использования

- **Версионный контроль** — коммитьте экспортированные файлы для отслеживания изменений переводов
- **Пайплайны деплоя** — экспортируйте перед деплоем, если нижестоящие инструменты ожидают файловые переводы
- **Резервные копии** — создавайте точечные снимки всех переводов
- **Другие инструменты** — экспортируйте для использования в сервисе управления переводами или CSV-импортёре

### Через фасад

```php
Lingua::syncToLocal();
```

## Обновление из Laravel Lang

Загружает последние строки переводов из экосистемы `laravel-lang` и синхронизирует их в базу данных. Полезно после обновления Laravel или добавления нового пакета с переводами.

```bash
php artisan lingua:update-lang
```

Выполняет `lang:update` (из `laravel-lang/common`) с последующим `lingua:sync-to-database`.

## Автоматизированные рабочие процессы синхронизации

### При деплое

Добавьте шаг после деплоя для поддержания синхронизации базы данных с закоммиченными файлами lang:

```bash
# В скрипте деплоя или CI/CD-пайплайне
php artisan lingua:sync-to-database
php artisan optimize:clear
```

### Через планировщик

Если ваша команда переводчиков редактирует файлы напрямую (а не через интерфейс), запланируйте периодическую синхронизацию:

```php
// routes/console.php
Schedule::command('lingua:sync-to-database')->hourly();
```

### При установке пакета

Когда вы выполняете `composer require` нового пакета с переводами, запустите:

```bash
php artisan lingua:update-lang
```

Это подхватит все новые строки из установленного пакета.

## Советы и подводные камни

::: tip Сохраняйте БД как источник истины
Считайте базу данных основным источником. Синхронизируйте в локальные файлы только когда нужны файлы (версионный контроль, деплой и т.д.). Избегайте прямого редактирования локальных файлов, пока БД в использовании — следующая sync-to-database перезапишет ваши правки, если ключи уже существуют.
:::

::: warning Локальные файлы и БД не синхронизированы
Если вы добавляете новые PHP-файлы локали вручную без запуска `lingua:sync-to-database`, новые ключи будут доступны только через файловый загрузчик (с более низким приоритетом, чем БД). Запустите sync для их правильного импорта.
:::

::: tip Полный цикл
Безопасный способ реорганизации переводов:
1. `lingua:sync-to-local` — экспортировать всё
2. Отредактировать файлы на диске
3. `lingua:sync-to-database` — повторно импортировать
:::
